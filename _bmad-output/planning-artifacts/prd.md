# Product Requirements Document - moonstonebaby-client

**Author:** wooa  
**Date:** 2026-01-21

---

## Success Criteria

### User Success

- **신뢰(즉시성/정확성)**: 아이(대상) 전환 및 기록 후 화면이 항상 즉시 올바른 데이터로 업데이트된다.
- **안심하고 지속 사용**: 가족이 일상적으로 계속 사용해도 불안(동기화/누락/권한 문제) 없이 운영된다.

### Business Success

- **가족 지속 사용(비공개)**: 공개 서비스가 아니라 “우리 가족만”을 위한 앱으로, 가족 구성원들이 꾸준히 사용한다.
- **개인 서버 운영 가능**: 개인 서버 기반으로 안정적으로 운영 가능하며, 관리 부담이 과도하지 않다(운영 방식은 추후 구체화).

### Technical Success

- **기록 유실 0**: 이벤트 기록/미디어 메타데이터의 생성/수정/삭제가 저장/동기화 과정에서 유실되지 않는다.
- **권한 누수 0**: 초대/권한 모델이 확실하며, 보조 사용자가 접근 권한 밖의 데이터(아이/앨범/일지)를 절대 볼 수 없다.

### Measurable Outcomes

- **아이 전환 정확성**: 아이 전환 시 잘못된 아이 데이터 노출이 발생하지 않는다.
- **동기화 무결성**: 여러 디바이스/계정에서 CRUD가 최종적으로 동일한 상태로 수렴하며, 충돌 시에도 유실이 없다.
- **접근 제어 검증**: 권한별로 접근 가능한 리소스가 명확히 분리되고, 링크 공유/초대 흐름에서도 권한 누수가 없다.

## Product Scope

### MVP - Minimum Viable Product

- 아이별 타임라인 앨범(사진/동영상)
- 이벤트 기록(육아일지) CRUD
- 다기기/다계정 동기화
- 초대/권한(RBAC/역할 기반 기본 읽기, 필요 시 추가 권한)
- 댓글/반응(보조 사용자 참여)

### Growth Features (Post-MVP)

- 성장/발달 차트
- 패턴 고도화(주/월 리포트, 비교 차트)
- 사진 자동 정리(태깅/분류)
- 데이터 내보내기/백업

### Vision (Future)

- “가족 데이터의 장기 자산화”: 아이 성장 스토리(사진+일지)가 장기간 축적되어도 검색/정리/이관이 쉬운 형태로 진화

## User Journeys

### Journey 1: 부모(엄마) - 일상 기록 여정 (Success Path)

**Opening Scene**: 엄마 민지(30세)는 쌍둥이(6개월)를 키우며, 아침에 첫 번째 아이(민수)의 기저귀를 갈고 기록하려고 앱을 연다.

**Rising Action**:
- 앱 첫 화면에서 아이별 타임라인을 확인한다.
- 민수를 선택하고, "기저귀 갈기" 이벤트를 기록한다.
- 기록 직후 화면이 민수의 타임라인으로 즉시 업데이트되어 방금 기록한 이벤트가 보인다.
- 오후에 두 번째 아이(민영)의 몸무게를 측정하고 기록한다.
- 저녁에 찍어둔 사진들을 업로드한다.
- 업로드한 사진에 대해 각각 아이를 태그한다(민수/민영 선택).
- 가능하면 얼굴 특징 기반으로 자동 태깅 제안이 표시되어, 확인만 하면 된다.
- 일자별 타임라인에서 오늘 기록한 모든 이벤트와 사진을 시간순으로 확인한다.

**Climax**: 아빠가 다른 기기에서 기록한 분유 이벤트가 실시간으로 동기화되어 보인다. "우리 둘 다 기록해도 꼬이지 않네!"

**Resolution**: 민지는 두 아이의 일일 패턴을 타임라인에서 한눈에 파악하고, 내일 수유/기저귀 타이밍을 예측할 수 있게 된다.

### Journey 2: 부모(부부) - 동기화 및 아이 전환 신뢰성 (Edge Case)

**Opening Scene**: 아빠 현우가 집에서 민수의 분유 이벤트를 수정하려고 한다. 동시에 엄마 민지가 회사에서 같은 이벤트를 수정하려고 한다.

**Rising Action**:
- 아빠가 먼저 수정을 완료한다(클라이언트 시간: 14:00).
- 엄마가 14:01에 수정을 완료한다.
- 서버는 두 요청을 받아 서버 타임스탬프를 기준으로 마지막 업데이트(엄마의 수정)를 반영한다.
- 아빠의 기기에 "다른 사용자가 수정했습니다. 최신 버전으로 업데이트되었습니다" 알림이 표시된다.
- 아빠는 이벤트 상세 화면에서 "변경 이력" 버튼을 눌러, 자신이 수정한 버전과 엄마가 수정한 최신 버전을 비교할 수 있다.

**Climax**: 네트워크가 끊긴 상태에서 아빠가 기저귀 이벤트를 기록한다. 앱은 로컬에 저장하고 "오프라인 모드 - 동기화 대기 중" 표시를 보여준다. 네트워크 복구 시 자동으로 동기화가 완료되고, 엄마의 기기에도 즉시 반영된다.

**오프라인 충돌 시나리오**:
- 아빠가 오프라인에서 민수의 몸무게를 7.5kg으로 수정한다.
- 엄마가 온라인에서 같은 이벤트를 7.8kg으로 수정한다.
- 네트워크 복구 후 아빠의 기기가 동기화를 시도한다.
- 서버 타임스탬프 기준으로 엄마의 수정(7.8kg)이 더 최신이므로, 아빠의 로컬 수정(7.5kg)은 덮어쓰기된다.
- 아빠는 이벤트 상세에서 "변경 이력"을 확인하여, 자신의 오프라인 수정(7.5kg)과 최종 반영된 값(7.8kg)을 모두 볼 수 있다.

**아이 전환 명시적 확인**:
- 민수 → 민영으로 전환 시, 화면 상단에 "민영의 타임라인으로 전환했습니다" 토스트 메시지가 2-3초간 표시된다.
- 동시에 타임라인과 일지 데이터가 즉시 민영의 것으로 업데이트되어, 잘못된 아이 데이터가 보이지 않는다.

**Resolution**: 부부는 네트워크 상태와 관계없이 안심하고 기록할 수 있으며, 동시 수정 시에도 데이터 유실 없이 마지막 수정이 반영되고, 변경 이력을 통해 이전 버전도 확인할 수 있다.

### Journey 3: 보조 사용자(보모) - 이벤트 기록 및 성장 앨범 구독

**Opening Scene**: 보모 김선생(40세)이 카카오톡으로 초대 링크를 받고 앱에 접속한다.

**Rising Action**:
- 링크를 클릭하여 앱에 접속한다.
- 첫 접속 시 "당신은 누구인가요?" 화면에서 "보모"를 선택한다.
- 부모(민지)가 보모의 권한을 "이벤트 기록 가능"으로 설정한다.
- 보모는 민수와 민영의 타임라인을 확인할 수 있고, 모든 이벤트 타입(기저귀/몸무게/분유/이유식)을 기록할 수 있다.
- 보육 중 민수의 기저귀를 갈고, 앱에서 즉시 기록한다.
- 기록한 이벤트가 부모의 기기에도 실시간으로 동기화되어 반영된다.

**Climax**: 보모가 기록한 이벤트에 대해 부모가 댓글로 "고생하셨어요!"라고 남긴다. 보모는 자신이 기록한 내용이 가족과 공유되고 있다는 것을 느낀다.

**Resolution**: 보모는 보육 중 발생하는 모든 이벤트를 기록할 수 있으며, 그 기록이 부모와 실시간으로 공유되어 협업이 원활해진다.

**권한 범위**: 보모는 모든 이벤트를 기록할 수 있으나, 수정/삭제는 부모만 가능하다(또는 부모가 별도 권한 부여 시 가능).

### Journey 4: 부모(관리자) - 가족 멤버 관리 및 동기화 실패 복구

**Opening Scene**: 엄마 민지가 새로 온 보모(김선생)를 가족 멤버로 추가하려고 한다.

**Rising Action**:
- 설정 메뉴에서 "가족 멤버 관리"를 선택한다.
- "멤버 초대" 버튼을 누르고, 카카오톡 링크를 생성한다.
- 링크를 카카오톡으로 보모에게 전달한다.
- 보모가 링크를 통해 접속하고, "보모" 관계를 설정한다.
- 민지가 보모의 권한을 "이벤트 기록 가능"으로 설정한다.

**동기화 실패 시나리오**:
- 네트워크가 불안정한 상태에서 민지가 민수의 분유 이벤트를 기록한다.
- 동기화가 실패하고, 화면 하단에 "동기화 실패 - 재시도 중..." 토스트 메시지가 표시된다.
- 앱은 주기적으로(예: 30초마다) 동기화를 재시도한다.
- 실패한 기록은 로컬에 저장되어 있으며, 온라인 상태에서 같은 항목의 최신 버전이 서버에 존재하면 로컬 버전을 덮어쓰기한다.
- 동기화 성공 시 "동기화 완료" 토스트가 표시된다.

**Resolution**: 민지는 가족 구성원을 유연하게 관리하며, 네트워크 문제가 있어도 기록이 유실되지 않고 자동으로 복구된다.

### Journey Requirements Summary

위 여정들이 드러내는 핵심 기능 요구사항:

1. **아이 선택 및 전환**: 명확한 UI, 토스트 메시지, 즉시 데이터 업데이트
2. **이벤트 기록**: 기저귀/몸무게/분유/이유식 등, 아이별 CRUD
3. **이벤트 변경 이력**: 모든 수정 이력을 추적하고, 사용자가 이전 버전을 확인할 수 있음
4. **사진 업로드 및 태깅**: 업로드 후 아이 태그, 얼굴 인식 기반 자동 태깅 제안
5. **다계정 동기화**: 부모 계정 간, 오프라인→온라인 동기화, 서버 타임스탬프 기반 충돌 해결
6. **동기화 실패 복구**: 토스트 메시지, 주기적 재시도, 로컬 저장 후 최신 버전으로 덮어쓰기
7. **초대 및 권한 관리**: 카카오톡 링크 기반 초대, 역할별 권한 설정(읽기/이벤트 기록/수정 삭제)
8. **보조 사용자 경험**: 보모 등도 모든 이벤트 기록 가능, 사진 댓글/반응

## Domain-Specific Requirements

### Technical Constraints

**플랫폼 요구사항**:
- **Expo 기반 크로스 플랫폼**: 모바일(iOS/Android)과 웹 클라이언트를 동시에 구성
- 모바일과 웹 간 동일한 기능 제공 및 데이터 동기화 일관성 유지
- 웹에서는 모바일 특화 기능(카메라, 푸시 알림 등)의 대체 UX 제공

### Privacy & Data Security

**개인정보 보호**:
- 아동 데이터 및 가족 공유 데이터의 프라이버시 보장
- 로컬 저장 시 데이터 암호화 고려
- 개인 서버 운영을 통한 데이터 소유권 강화

### Server Infrastructure

**개인 서버 운영**:
- 서버 보안 및 백업 전략 수립 필요
- 데이터 접근 제어 및 권한 관리 시스템 구축
- 운영 부담 최소화를 위한 자동화 고려

## Innovation & Novel Patterns

### Detected Innovation Areas

#### 1. 얼굴 인식 기반 자동 태깅 (AI-Powered Face Recognition Tagging)

**혁신적 요소**:
- 쑥쑥찰칵의 AI 사진 자동 정리는 날짜/월령별 분류에 초점이 있으나, 본 제품은 **얼굴 특징 기반으로 아이를 자동 인식하여 태그**하는 기능을 제공
- 부모가 수동으로 아이를 선택할 필요 없이, 업로드한 사진에서 쌍둥이/다자녀를 자동으로 구분하여 태그 제안
- MVP에서는 Growth Feature로 분류하되, 수동 태깅과 함께 제공

**기술적 고려사항**:
- **클라이언트 처리 가능성**: 얼굴 인식을 클라이언트(로컬)에서 처리하는 것이 기술적으로 가능한지 검토 필요
  - 장점: 프라이버시 강화, 오프라인 처리 가능, 서버 부하 감소
  - 고려사항: 모바일 기기 성능, 모델 크기, 정확도
- **태깅 동기화**: 얼굴 인식으로 생성된 태그가 다른 사용자(부모, 보조 사용자)에게도 전파되어야 함
  - 클라이언트 처리 시: 태그 결과를 서버에 동기화하여 모든 사용자에게 반영
  - 서버 처리 시: 서버에서 처리 후 결과를 모든 클라이언트에 전파

**사용자 가치**:
- 수동 태깅 대비 **상당한 시간 절약** (특히 다자녀 환경에서)
- 사진 업로드 후 즉시 태그 제안으로 사용자 편의성 향상

**차별화 포인트**:
- 경쟁사들은 주로 수동 태깅 또는 날짜 기반 자동 분류에 의존
- 본 제품은 다자녀 환경에서 각 아이를 자동으로 구분하여 태그하는 AI 기능 제공

#### 2. 사진+일지 통합 타임라인 경험 (Unified Timeline Experience) - MVP 핵심 혁신

**혁신적 요소**:
- 사진 공유와 육아일지(이벤트 기록)를 **하나의 시간순 타임라인에서 통합**하여 제공
- 경쟁사들은 대부분 사진 공유 앱과 일지 앱이 분리되어 있거나, 통합되어 있어도 타임라인 통합이 약함
- 사용자가 아이의 성장 스토리를 사진과 일지 데이터를 함께 시간순으로 추적할 수 있음

**사용자 가치**:
- **성장 상태 통합 확인**: 사진과 일지 데이터를 함께 보며 아이의 성장 상태를 종합적으로 파악
- **평균 대비 차이 분석**: 통합 데이터를 통해 아이의 성장 패턴이 평균 대비 어떤 차이가 있는지 확인 가능
- 감정적 가치: 아이의 성장 스토리를 한눈에 볼 수 있어서 감동적

**차별화 포인트**:
- 쑥쑥찰칵: 사진 중심, 일지 기능 약함
- 베이비타임: 일지 중심, 사진 통합 약함
- 본 제품: 사진과 일지를 동등하게 타임라인에 통합하여 "스토리텔링" 경험 제공

**MVP 우선순위**: 통합 타임라인이 MVP에서 더 중요한 핵심 차별화 요소

### Market Context & Competitive Landscape

**경쟁사 분석**:
- 쑥쑥찰칵: AI 사진 자동 정리(날짜/월령별), 사진 중심, 일지 기능 제한적
- 베이비타임: 일지 데이터 분석 강점, 사진 통합 약함
- 맘스다이어리: 사진+일지 결합이지만 타임라인 통합 경험이 약함

**시장 기회**:
- 통합 경험을 원하는 사용자 니즈가 있으나, 현재 시장에서 완전한 통합 솔루션이 부족
- 다자녀 가정의 증가와 함께 자동 태깅 니즈가 증가할 것으로 예상

### Validation Approach

**얼굴 인식 자동 태깅**:
- MVP에서는 Growth Feature로 분류하되, 수동 태깅과 함께 제공
- 클라이언트 처리 가능성 검토 및 태깅 동기화 메커니즘 설계
- 얼굴 인식 정확도를 측정하고, 사용자 피드백을 통해 개선
- 쌍둥이/다자녀 환경에서 얼굴 구분 정확도가 핵심 지표
- 사용자 시간 절약 효과 측정 (수동 태깅 대비)

**통합 타임라인 경험**:
- 사용자 여정에서 "아하!" 순간이 발생하는지 측정
- 타임라인에서 사진과 일지를 함께 확인하는 사용자 비율 측정
- 성장 상태 통합 확인 및 평균 대비 차이 분석 기능의 사용 빈도 측정
- 경쟁사 대비 사용자 만족도 비교

### Risk Mitigation

**얼굴 인식 관련 리스크**:
- **정확도 문제**: 쌍둥이/다자녀에서 얼굴 구분이 어려울 수 있음
  - 완화: MVP에서는 제안 형태로 제공하고, 사용자 확인을 통해 정확도 보완
  - Fallback: 수동 태깅을 항상 제공하여 자동 태깅 실패 시에도 사용 가능

- **프라이버시 우려**: 얼굴 인식 데이터 처리에 대한 사용자 우려
  - 완화: 클라이언트 처리 우선 검토, 서버 전송 시 암호화, 사용자 동의 명확히

- **태깅 동기화 복잡성**: 클라이언트 처리 시 태그 결과를 모든 사용자에게 동기화하는 메커니즘 필요
  - 완화: 서버에 태그 메타데이터만 저장하고, 모든 클라이언트에 전파하는 동기화 로직 설계

**통합 타임라인 관련 리스크**:
- **성능 문제**: 사진과 일지 데이터를 함께 로드할 때 성능 저하 가능
  - 완화: 지연 로딩, 페이지네이션, 캐싱 전략 수립
  - Fallback: 사진/일지 필터링 기능 제공하여 선택적 표시 가능

- **복잡성 증가**: 통합 경험이 오히려 복잡해 보일 수 있음
  - 완화: 직관적인 UI/UX 설계, 사용자 테스트를 통한 검증

### Competitive Advantage Sustainability

**현재 단계**:
- 제품 개발 및 사용자 피드백 수집을 통해 경쟁 우위 지속성 검토
- 초기 MVP에서는 통합 타임라인 경험에 집중하여 차별화 확보
- 얼굴 인식 자동 태깅은 Growth Feature로 개발하며, 기술적 우위 확보 방안 모색
